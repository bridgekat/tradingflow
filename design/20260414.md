# Design Notes — 2026-04-14

## 1. Prediction Horizon: Sampling Frequency, Not Trading Frequency

### Decision

Predictors target **1-period (sampling-frequency) returns**, not K-period returns aligned with the rebalance interval.

### Rationale

Prediction is estimation. The quality of a statistical estimate depends on the number of independent observations. With daily sampling:

- **Daily target**: each trading day is one observation. 250 observations/year.
- **K-period target** (e.g. K=120): non-overlapping windows give ~2 independent observations/year. Overlapping windows are highly correlated and don't help much. You cannot fit a model with statistical confidence.

The rebalance interval K is a **trading constraint** (driven by transaction costs), not a property of the signal. A value factor has daily predictive power — the IC is just tiny (~0.002) and noisy. Over K days, the signal accumulates and noise partially cancels, but this is the same daily signal viewed through a lower-noise lens, not a fundamentally different prediction.

### Consequences

- `MeanPredictor` and `VariancePredictor` should compute forward returns as `prices[t+1] / prices[t] - 1` (1-period), not `prices[t+K] / prices[t] - 1`.
- `rebalance_period` controls how often the predictor refits and the portfolio rebalances, but does not affect the target variable.
- K-period portfolio statistics (expected return, variance) are derived from the daily predictions: `μ_K ≈ K · μ_daily`, `Σ_K ≈ K · Σ_daily` (exact under independence; approximate with autocorrelation).
- Factor evaluation via `InformationCoefficient` naturally evaluates daily IC when the prediction series is recorded at sampling frequency.
- Covariance evaluation via `MinimumVariance` accumulates daily GMV returns between rebalance points, which is now consistent: the predicted Σ is a daily covariance, and the realized variance is measured from daily returns.

## 2. Compounding and the Kelly Criterion

### Problem

Optimizing daily simple returns does not maximize multi-period wealth. Portfolio value after K days is `∏(1 + wᵀr_t)`, which is multiplicative — variance reduces compound growth even when expected daily return is positive.

### Resolution

This is already handled by mean-variance optimization. The second-order Taylor expansion of `ln(1 + r_p)` around `r_p = 0` gives:

    ln(1 + r_p) ≈ r_p − ½ r_p²

    E[ln(1 + r_p)] ≈ μ_p − ½(σ²_p + μ²_p)

For daily returns, μ_p is on the order of 10⁻⁴, so μ²_p ≈ 10⁻⁸ is negligible compared to σ²_p ≈ 10⁻⁴. Dropping the μ²_p term gives the standard approximation:

    E[ln(1 + r_p)] ≈ μ_p − ½ σ²_p

This is Markowitz with risk aversion δ = ½ (the Kelly criterion), maximizing long-run compound growth. The full expansion gives a slightly different objective with an additional penalty on squared expected return, but the difference is immaterial at the daily scale.

No new mechanism is needed. Predicting daily μ and Σ is sufficient; the compounding problem is solved at the portfolio optimization layer by choosing the appropriate risk aversion. δ = 0.5 gives Kelly-optimal growth; other values of δ trade off growth rate against drawdown tolerance.

## 3. Rank-Based / Gaussianized Features in Linear Regression

### Motivation

Since factor evaluation uses RankIC (Spearman) rather than Pearson IC, there's a natural question: should the linear regression predictor itself operate on rank-transformed inputs? Rank / quantile transformation of cross-sectional factors is a very common technique in equity factor research.

### Why rank features (X)

- **Outlier robustness**: factors like book-to-price, earnings yield, and momentum have heavy tails. A single extreme observation can dominate OLS. Ranks are bounded in [1, N].
- **Scale invariance**: multi-factor models become trivial to stack when every column lives in the same range. No need to z-score or winsorize each factor separately.
- **Captures monotonic non-linearity**: if returns are monotone but not strictly linear in factor value (common for value/quality), rank regression captures more signal than OLS on raw values.
- **Metric–objective alignment**: if evaluation uses RankIC, training on ranks optimizes the same thing being measured.

### Common variants (in increasing sophistication)

1. Pure ranks: `1, 2, ..., N`.
2. Normalized ranks: `(rank − 0.5) / N ∈ (0, 1)` — uniform on the cross-section.
3. Gaussianized ranks: `Φ⁻¹((rank − 0.5) / N)` — each factor column looks N(0, 1) cross-sectionally. Standard enough that pandas / sklearn / scipy all provide it (`QuantileTransformer(output_distribution="normal")`).
4. Decile / quintile dummies — extreme discretization; used in long-short portfolio construction.
5. Pairwise rank losses (LambdaRank, RankNet) — full learning-to-rank; used by some ML-heavy quant funds.

### Asymmetry: don't rank Y (returns)

The case against ranking the target variable:

- **Returns have a meaningful scale** (percentage points). A 10% return is meaningfully different from 1%.
- **Markowitz optimization needs μ in return units**. Training on ranked Y produces predictions in rank space; the portfolio optimizer can't sensibly trade off `wᵀμ` against `wᵀΣw` because units don't match.
- **Magnitude information is genuinely useful** and gets discarded by ranking.

Typical industry practice: **winsorize Y** (clip at ±5σ or a raw threshold) and then fit OLS. Keeps magnitudes for the bulk of the distribution while taming outliers. Alternatives include Huber loss or quantile regression. Pure rank-on-rank is mainly seen in pipelines whose downstream is also rank-based (e.g., long top-decile minus bottom-decile, equal-weighted).

### Recommendation

- **X (features): rank-transform or Gaussianize cross-sectionally.** High upside, low downside. Becomes more valuable as the number of features grows.
- **Y (returns): keep raw, optionally winsorize.** Preserves magnitudes for downstream mean-variance optimization.

### Planned operators

- `RankCS(array)` — cross-sectional rank-transform (or Gaussianize via inverse normal CDF). Applied to each factor before `Stack`-ing them into `stacked_features`.
- `Winsorize(array, lower_quantile, upper_quantile)` — cross-sectional clip at given quantiles. Applied to returns upstream of the predictor if needed.

Both are straightforward Python `Map`-style operators over an `Array(N,)`. The existing `_rankdata` helper in `information_coefficient.py` already implements the per-tick rank logic with average tie-breaking and could be lifted into a shared utility.
