# Design Notes — 2026-04-08 to 2026-04-13

## 1. Architectural Review and Notification Semantics

### Problem: Stack semantics for suspended stocks

When a stock is suspended (no data for a day), the `Stack` operator copies its stale value from the previous day. Combined with `ForwardFill`, suspended stocks appear with yesterday's full OHLCV (including nonzero volume), which is incorrect. The correct fill would be O = H = L = C = previous close, V = 0.

The root cause: Stack reads all inputs on every trigger but cannot distinguish "this input updated today" from "this input holds a stale value." The `Notify` mechanism (at the time) only provided a per-cycle boolean flag that was too ephemeral and too coarse for this purpose.

### Solution: enriched Notify with per-input produced tracking

Replaced the graph's `pending: Vec<bool>` and `produced: Vec<bool>` with `incoming: Vec<Vec<usize>>`, where each entry records *which input positions* triggered the operator. Trigger edges now carry `(downstream_node, input_position)` pairs.

The `Notify` context exposes two views:

- `produced() -> &[usize]` — list of input positions that produced (O(n_messages) iteration).
- `input_produced() -> Box<[bool]>` — per-position booleans, computed lazily on call (O(1) indexing). Construction of `Notify` is zero-cost.

This enables two complementary operator semantics:

- **Time-series semantics**: operator ignores `Notify`, reads latest values from all inputs. Most operators use this.
- **Message-passing semantics**: operator inspects `Notify` to distinguish updated inputs from stale ones. Used by `ForwardAdjust`, and in the future by a Notify-aware `Stack` variant for the suspended-stock problem.

### Clock-triggerable guard

Operators that rely on message-passing must not be clock-triggered (they'd miss messages between clock ticks). Added `is_clock_triggerable()` method to the `Operator` trait (default `true`). The runtime panics at registration time if a non-clock-triggerable operator is given a clock trigger.

## 2. Predictor Refactoring: Series Inputs

### Problem: unbounded memory growth in predictors

`MeanPredictor` and `VariancePredictor` accumulated `features_list` and `prices_list` internally on every tick, growing without bound.

### Solution: external accumulation via Record operators

Predictors now take `Series` inputs (produced by upstream `Record` operators) instead of accumulating internally. The `compute()` method reads history via `SeriesView` indexing. This:

- Removes unbounded state from predictors.
- Separates accumulation from fitting/prediction (enables future truncation operators).
- Makes the pipeline more composable (PCA or other transforms can be inserted between `Record` and predictor).

### Subclass kwargs forwarding

Predictor and portfolio subclasses forward base-class config parameters (`rebalance_period`, `max_samples`, `min_samples`, `trading_start`) via `**kwargs`, keeping handle parameters and subclass-specific parameters explicit.

## 3. NaN-Robust Portfolio Construction

### Mean-only portfolios

`MeanPortfolio.compute()` masks stocks with non-finite predicted returns before passing to `positions_fn`. Subclass functions receive a clean sub-array.

### Mean-variance portfolios

`MeanVariancePortfolio.compute()` masks stocks with non-finite predicted returns or non-finite diagonal covariance entries. Raises `ValueError` if the sub-covariance matrix of remaining stocks still contains non-finite entries.

## 4. Per-Stock Minimum Sample Filtering

Added `min_samples` parameter to both `MeanPredictor` and `VariancePredictor`. After computing forward returns for the sampled time rows, the base class counts per-stock valid observations (finite features AND finite return). Stocks below the threshold are excluded from `fit_fn`/`predict_fn` entirely and receive NaN in the output. Downstream NaN-robust portfolio constructors assign them zero weight.

## 5. Training/Trading Period Separation

### trading_start parameter

Added `trading_start: datetime64 | None` to predictors, portfolio constructors, and traders (converted to nanoseconds internally via `coerce_timestamp`).

- **Predictors**: suppress output (return `False`) before `trading_start`, but continue counting ticks so rebalance timing is preserved.
- **Portfolios**: return `False` before `trading_start` (safety net).
- **Traders/Benchmarks**: skip rebalance before `trading_start`, but still track dividend adjustments and output current portfolio value.

### Example CLI changes

Both backtesting examples gained `--data-begin` (default 1990-01-01) for source time range. `--begin` is now the trading start date. Progress bars show separate "Loading samples" and "Running strategy" phases.

## 6. Factor Evaluation Metrics

### AlphaEval framework (from arXiv 2508.13174)

Five backtest-free dimensions for evaluating formula alpha signals:

| Metric | Formula | Measures |
|--------|---------|----------|
| **PPS** (Predictive Power Score) | `β · IC + (1-β) · RankIC` | Combined linear and monotonic predictive power |
| **RRE** (Relative Rank Entropy) | `mean_t 1/(1 + KL(S_t ∥ S_{t-1}))` | Temporal stability of cross-sectional rankings |
| **PFS** (Perturbation Fidelity Score) | `min(SpearmanCorr(S, S'_gaussian), SpearmanCorr(S, S'_t-dist))` | Robustness to input noise |
| **Logic Score** | LLM-rated financial interpretability | Qualitative plausibility |
| **DH** (Diversity Entropy) | Normalized spectral entropy of alpha covariance matrix | Redundancy / multicollinearity of an alpha set |

Traditional backtesting metrics defined in the paper's appendix: ICIR, RankICIR, AR (annualized return from long-short portfolio), SR (Sharpe ratio), Turnover (L1 norm of weight changes), MaxDD (maximum drawdown).

### Key limitation of AlphaEval

The paper fixes ΔT = 1 (single-day forward returns) and never varies the prediction horizon. A factor with low daily IC but strong monthly IC would be dismissed.

## 7. IC: Interpretation and Practical Thresholds

### IC computation structure

For a factor with score matrix S(T,N) and forward returns y(T,N,ΔT):

- A single sample is the triplet (time, forward_period, symbol).
- Cross-sectional IC at time t: `IC(t, ΔT) = Corr_n(S_{t,n}, y^{ΔT}_{t,n})`.
- IC decay curve: `IC̄(ΔT) = mean_t IC(t, ΔT)` — mean cross-sectional IC as a function of horizon.
- Cumulative IC: `cumsum_t IC(t, ΔT)` — diagnostic plot, not a meaningful standalone quantity.
- ICIR: `mean_t IC(t, ΔT) / std_t IC(t, ΔT)` — consistency of the signal.

The IC decay curve cannot be derived from IC at a single horizon. Its shape depends on the factor's signal structure (persistent drift, single future event, mean-reverting, delayed reaction). There is no universal √ΔT scaling law. Evaluating a factor at the horizon you actually trade at requires running the experiment at that horizon.

### Rough IC thresholds (daily cross-sectional, equities)

| IC range | Interpretation |
|----------|---------------|
| > 0.05 | Very strong — check for lookahead bias |
| 0.03 – 0.05 | Strong single factor |
| 0.01 – 0.03 | Decent for multi-factor combination |
| 0.005 – 0.01 | Weak but potentially usable |
| < 0.005 | Noise |

ICIR > 0.5 matters more than high IC alone. IC × √(periods per year) is a rough horizon-invariant comparison measure.

### IC thresholds scale with horizon

Monthly IC is higher than daily IC for factors with persistent signals, but with fewer independent observations per year. The thresholds cannot be converted across horizons without knowing the factor's signal structure.

## 8. Evaluating Covariance Predictors

Unlike mean prediction (where the ground truth is a single observable return), the "true" covariance matrix is never directly observable — only estimable from a window of future returns.

Approaches and their problems:

| Approach | What it tests | Problem |
|----------|--------------|---------|
| GMV portfolio realized variance | Covariance via min-variance weights | Concentrates on least-volatile stocks, ignores most of the matrix |
| Realized log-likelihood | Full matrix via `log det(Σ) + r'Σ⁻¹r` | Requires T > N for invertibility; ground truth is itself an estimate |
| Portfolio variance ratio | Specific portfolio directions | Only tests the directions you choose |
| Variance IC | Diagonal only: `Corr_n(diag(Σ_pred), realized_var_n)` | Ignores off-diagonal structure |

Practical conclusion: covariance prediction quality is best evaluated via end-to-end portfolio metrics (Sharpe, Drawdown) rather than isolated accuracy measures. Variance IC is the most tractable direct metric.

## 9. Log Returns vs. Simple Returns

| Property | Simple return `r = P₁/P₀ - 1` | Log return `R = ln(P₁/P₀)` |
|----------|-------------------------------|----------------------------|
| Cross-sectional aggregation | Additive: `r_p = Σ wᵢrᵢ` (exact) | Not additive |
| Time-series aggregation | Multiplicative: `Π(1 + rᵢ) - 1` | Additive: `Σ Rᵢ` |

The tension: no return definition is additive in both dimensions.

### Decision for our pipeline

Use **simple returns** throughout. Rationale:

- Portfolio construction does cross-sectional weighted averaging, which requires simple returns.
- The predictor's job is to predict which stocks will earn more in the next period; simple returns preserve the correct cross-sectional ordering.
- Log returns would implicitly penalize volatile stocks in the mean prediction (a stock that doubles or halves with 50/50 probability has 25% simple expected return but 0% log expected return). This volatility penalty belongs in the optimizer, not the predictor.

### Kelly criterion and risk aversion

Maximizing expected log-wealth `E[ln(Σ wᵢ(1 + rᵢ))]` requires the full joint distribution. Under a second-order Taylor approximation (valid for small, diversified returns):

$$E[\ln(1 + r_p)] \approx \mu_p - \frac{1}{2}\sigma_p^2$$

This is Markowitz with risk aversion δ = 1/2, using simple returns for both μ and Σ. Our existing pipeline already supports this — the insight is that δ = 0.5 has a principled interpretation as the Kelly-optimal risk aversion.
