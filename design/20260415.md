# Design Notes — 2026-04-15

This session completed two major architectural refactors:

1. **Recursive `InputTypes` trait** — operator inputs become a fully nested tree (leaves × tuples × dynamic slices) instead of two mutually-exclusive top-level shapes.
2. **Module reorganisation** — `src/types.rs` → `src/data/` (with `array`, `series`, `time`, `inputs` as submodules).

Plus one interstitial refactor that was squashed into the above:

3. **Stream-based cursor traversal** — `write_type_ids` / `refs_from_flat` / `produced_from_flat` now thread a `FlatRead` / `FlatWrite` cursor through the recursion instead of re-computing sub-tree arities at each level. Traversal cost drops from O(depth × leaves) to O(total_leaves).

## 1. Recursive `InputTypes`

### Problem

The old trait split operator inputs into two mutually-exclusive forms: fixed-length heterogeneous tuples (arities 1–12) and a single variable-length homogeneous slice `[T]`. These cannot compose. An operator that wants "slice of data **plus** a clock" (Stack + clock) has no natural expression — we had been faking this with a `trigger: Option<Handle<()>>` sideband and `Const(NaN)` filler to smuggle clock ticks into the data-input channel. The sideband was an increasingly obvious ad-hoc patch.

Meanwhile, three goals kept pulling against each other:

1. **Arbitrary input/output value types** — any `T: Send + 'static` should flow through a node.
2. **Multi-input declarations** — heterogeneous tuples OR homogeneous dynamic slices.
3. **Clocks as first-class inputs** — no sideband, just `()`-valued inputs.

Requirement 1 says `(A, B)` should be a valid value type flowing through a node. Requirement 2 says `Inputs = (A, B)` means "two inputs" — which conflicts with the first reading. These felt exclusive.

### Resolution: the `Input<T>` wrapper

**Key move**: separate *tree branches* from *tree leaves* at the type level via distinct type constructors.

- **Branches**: bare tuples `(A, B, …)` and a new sized wrapper `Slice<T>`.
- **Leaves**: a single zero-sized wrapper `Input<T>` that carries any value type `T: Send + 'static`.

The wrapper is invisible at runtime — `Input<T>::Refs<'a> = &'a T`, so compute bodies see `&T` directly. It exists only at the type level, where it forces the question "is this a branch or a leaf?" to be answered syntactically rather than by a blanket-vs-direct-impl coherence dance.

```rust
type Inputs = (Input<Array<f64>>, Input<Series<f64>>);       // two inputs
type Inputs = Slice<Input<Array<f64>>>;                       // Stack-like
type Inputs = (Slice<Input<Array<f64>>>, Input<()>);          // Stack + clock
type Inputs = Slice<(Input<Array<f64>>, Input<Array<f64>>)>;  // slice of pairs
type Inputs = Input<(A, B)>;                                  // one leaf whose value IS a tuple
```

All three goals are now satisfied simultaneously: `Input<T>` wraps arbitrary value types (including tuple-typed ones); tuples and `Slice` are the two branching constructs; clocks are just `Input<()>` leaves.

The convention: `Operator::Inputs` is always an `InputTypes` tree — a tuple (including 0-tuple `()` for no inputs), a `Slice<T>`, or an `Input<T>`. No more `(T,)`-wrapping single inputs unless explicitly desired.

### Coherence

With `Input<T>`, `Slice<T>`, and tuples as distinct type constructors, the trait impls are syntactically disjoint:

```rust
impl<T> InputTypes for Input<T> { ... }
impl<T> InputTypes for Slice<T> { ... }
impl InputTypes for () { ... }
impl<A, B> InputTypes for (A, B) { ... }
// ... tuples up to arity 12
```

Rust's coherence checker is syntactic, so these are all fine. No marker-trait tricks, no specialisation, no blanket-impl workarounds. This was the earlier failure mode of the `InputType` / `InputTypes` / `InputTypeOrTypes` three-trait sketch: the compiler couldn't rule out that tuples might one day impl the leaf marker.

### `Slice<T>` vs `[T]`

We initially wrote the slice branch as `[T]` directly. This breaks when the slice appears in any non-last tuple position, because Rust requires all fields of a tuple type to be `Sized` (except the last, per recent ABI rules). `([T], U)` is illegal at the type-formation level.

The sized `Slice<T>` wrapper solves this at zero runtime cost (`PhantomData`). Everything works as expected in any position.

### Shape, Refs, Produced

`InputTypes` gains three associated types, each mirroring the input tree:

- `Shape: Send + 'static` — runtime description. For leaves it's `()`; for tuples a tuple of child shapes; for `Slice<T>` a `SliceShape<T>` containing `Box<[T::Shape]>` plus a cached offsets prefix-sum table.
- `Refs<'a>` — nested reference view. `Input<T>::Refs = &T` (wrapper invisible); `(A, B)::Refs = (A::Refs, B::Refs)`; `Slice<T>::Refs = SliceRefs<T>` (zero-allocation lazy view with O(1) `get(i)` via the offset table).
- `Produced<'a>` — nested boolean view mirroring `Refs`. Opt-in: operators that don't care never touch it.

`Notify` itself stays non-generic and flat. `input_produced()` keeps its current allocating-on-demand behaviour. If an operator wants a nested produced view, it calls `I::produced_from_flat(&notify.input_produced(), &shape)` explicitly. Notify's design is orthogonal to the input tree — keeping them independent avoided a wider-scoped churn.

### `ErasedOperator` carries the shape

`ErasedOperator` gains a `shape: Box<dyn Any + Send>` field. The `ComputeFn` and `InitFn` signatures grow a `&(dyn Any + Send)` shape argument; the monomorphised `erased_compute_fn<O>` downcasts to `<O::Inputs as InputTypes>::Shape` and uses it to drive `refs_from_flat`. The `Any` downcast compiles to a single TypeId comparison.

`OperatorState` owns the shape post-`init` and forwards it to every `compute` call. No shape field on `Node`; no shape lookup on the hot path beyond the pointer deref.

### `InputTypesHandles` mirrors the structure

A parallel trait at the scenario layer. For each tree shape, it defines `Handles` (the user-facing input tuple shape), plus cursor-style `write_node_indices` and a `shape(&handles) -> Self::Shape` method to build the runtime shape.

- `Input<T>::Handles = Handle<T>`.
- `(A, B)::Handles = (A::Handles, B::Handles)`.
- `Slice<T>::Handles = Vec<T::Handles>`.

Users don't see this machinery — they still call `sc.add_operator(op, (ha, hb, vec_of_handles), trigger)` as before.

### `FlatShapeFromArity` for the Python bridge

The type-erased bridge path doesn't know `O::Inputs` at compile time, so it can't build a concrete `Shape`. Most bridge-registered operators have flat shapes — tuples of `()`, or `SliceShape<Input<T>>` with unit element shapes. A helper trait:

```rust
pub trait FlatShapeFromArity: Sized {
    fn flat_shape_from_arity(arity: usize) -> Self;
}
```

is implemented for `()`, tuples-of-units up to arity 12, and `SliceShape<T>` where `T::Shape = ()`. `ErasedOperator::from_flat_operator<O>(op, arity)` uses it to build the shape from just the arity. The bridge calls this; the typed path (`Scenario::add_operator`) builds the shape via `InputTypesHandles::shape(handles)` instead.

## 2. Stream-based Cursor Traversal

### Problem

The first implementation of the recursive trait computed sub-tree arity once per recursion level to slice the flat buffer:

```rust
// OLD tuple impl
let mut offset = 0;
let n = A::arity(&shape.0);
let refs_a = unsafe { A::refs_from_flat(&ptrs[offset..offset+n], &shape.0) };
offset += n;
let n = B::arity(&shape.1);
let refs_b = unsafe { B::refs_from_flat(&ptrs[offset..offset+n], &shape.1) };
```

On a nested tree, `A::arity` itself recurses. For a right-heavy tree of depth d, total work is `∑ₖ O(k) = O(d²)` — quadratic.

### Resolution: cursors

Introduce two single-pass cursors:

```rust
pub struct FlatRead<'a, T>  { buf: &'a [T],     idx: usize }
pub struct FlatWrite<'a, T> { buf: &'a mut [T], idx: usize }
```

with `.take(n) -> &'a [T]`, `.next() -> T`, `.push(v)`, `.remaining() -> usize`.

The trait methods thread the cursor through the recursion. Tuple impls become a direct repetition with no bookkeeping:

```rust
// NEW tuple impl
fn refs_from_flat<'a>(reader: &mut FlatRead<'a, *const u8>, shape: &'a Self::Shape) -> Self::Refs<'a> {
    (
        unsafe { A::refs_from_flat(reader, &shape.0) },
        unsafe { B::refs_from_flat(reader, &shape.1) },
    )
}
```

Each leaf consumes exactly one slot via `reader.next()`. `Slice<T>` consumes its whole sub-buffer in one `reader.take(shape.total_arity())` call — O(1) from the cached prefix-sum — and hands the sub-slice to `SliceRefs`. No per-child arity recomputation anywhere in the traversal.

Total traversal cost is now **O(total_leaves)**, linear regardless of depth.

### `arity` is kept

`InputTypes::arity(&shape)` still exists — the caller needs it once at operator registration to pre-size the flat `TypeId` / node-index buffers before handing them to `FlatWrite`. One O(total_leaves) walk at registration time, not on the hot path.

### Applied to `InputTypesHandles` too

`InputTypesHandles::write_node_indices` got the same treatment. Before: manual offset math with per-child arity calls. After: cursor threaded through the recursion.

### Unit-test sanity check

The `types::tests` helpers wrap each call in a cursor and assert `reader.remaining() == 0` / `writer.remaining() == 0` after the trait method returns. This catches impl bugs that consume too few or too many slots — a useful safety net during the migration.

## 3. Module Reorganisation

### Before

```
src/
  array.rs      -- Array
  series.rs     -- Series
  time.rs       -- Instant, Duration
  types.rs      -- Scalar, PeekableReceiver, Notify, InputTypes machinery
  operator.rs
  source.rs
  operators/
  scenario/
  sources/
  bridge/
  utils/
  lib.rs
```

`types.rs` was mixed: primitive trait (`Scalar`), concurrency primitive (`PeekableReceiver`), signalling primitive (`Notify`), and a large InputTypes machinery. Meanwhile `array`, `series`, `time` sat at the crate root as siblings of everything else.

### After

```
src/
  data/
    mod.rs      -- Scalar, PeekableReceiver, Notify + submodule decls + re-exports
    inputs.rs   -- InputTypes, Input, Slice, SliceShape, SliceRefs, SliceProduced,
                   FlatRead, FlatWrite, FlatShapeFromArity
    array.rs    -- Array
    series.rs   -- Series
    time.rs     -- Instant, Duration, utc_to_tai, tai_to_utc
  operator.rs
  source.rs
  operators/
  scenario/
  sources/
  bridge/
  utils/
  lib.rs
```

Grouping criterion: `data/` holds everything that *describes values and their types*. Operators / sources / scenarios that *consume* those values sit outside. The InputTypes machinery moved into its own submodule (`data::inputs`) since it's a big self-contained body of trait impls, macros, and helper types.

### Path migration

Top-level re-exports in `lib.rs` are unchanged (`crate::Array`, `crate::Input`, `crate::Instant`, etc. all still work), so user code is unaffected. Internally, every `crate::{types,array,series,time}::X` became `crate::data::X`. The sweep touched ~50 files and was handled by `sed`.

## 4. What Didn't Change

- `Notify` stays non-generic with allocating `input_produced()`. Nested-produced is opt-in via `I::produced_from_flat(&notify.input_produced(), &shape)`.
- The existing `trigger: Option<Handle<()>>` sideband on `Scenario::add_operator` stays. It's redundant now that clocks can be first-class inputs via `Input<()>`, but removing it is a separate follow-up.
- `is_clock_triggerable` / the message-passing operator gate stays.
- No operator transformers (`Clocked<O>`) introduced.
- Python-side operators unchanged.

## 5. Migration Summary

Breaking changes at the Rust operator authoring level:

- Every `type Inputs = (Array<T>, Series<T>)` now reads `type Inputs = (Input<Array<T>>, Input<Series<T>>)`. Mechanical wrap.
- `type Inputs = [Array<T>]` now reads `type Inputs = Slice<Input<Array<T>>>`. Compute bodies get a `SliceRefs<'_, Input<Array<T>>>` and access via `.get(i)` / `.iter()` instead of slice indexing. The shared `interleaved_copy` helper for `Concat` / `Stack` now takes `impl IntoIterator<Item = &Array<T>>`.
- Compute body parameter types *don't* change for tuple inputs, because `Input<T>::Refs<'a> = &'a T` — the wrapper is invisible past the trait system.

No Python-side changes required.

## 6. Verification

After both refactors:

- `cargo test --lib --no-default-features` — 200 passed, 0 failed (up from 187 before the refactor due to ~13 new `InputTypes` round-trip tests).
- `cargo build --features python --lib` — clean (one unrelated pyo3 deprecation warning in `NativeNotify::downcast`).
- `cargo build --benches --no-default-features` — clean.
- `grep 'Box<\[&' src/operators/` — no matches; confirms slice inputs no longer allocate a `Box<[&T]>` per compute call.
- `grep 'crate::\(types\|array\|series\|time\)::' src/` — no matches; module reorg is complete.

## 7. Follow-ups (not done this session)

- **Deprecate `trigger` / `is_clock_triggerable`** — superseded by `Input<()>` as a regular input slot. A later pass can remove the sideband and update existing users to pass clocks as real inputs.
- **`NativeNotify::input_produced` allocation** — still builds a `PyList` per call on the Python path. Moving to a borrowed-buffer representation is a Phase 2 concern.
- **Operator transformers** (`Clocked<O>`, `make_clocked`) — deferred; the current clock-as-input story covers the use cases that motivated them.
